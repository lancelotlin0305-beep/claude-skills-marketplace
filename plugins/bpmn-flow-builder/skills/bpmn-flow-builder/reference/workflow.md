# 工作流程規範(bpmn-flow-builder)

版面、繞線、閘道、分區、驗證等**製圖慣例**在 `conventions.md`;本檔規範**流程與交付**。

## 交付物(每張圖固定 6 檔)

| # | 檔名 | 用途 |
|---|------|------|
| 1 | `{圖名}_{版號}.drawio`(預設;明講 bpmn.io 才改 `.bpmn`) | 可編輯圖檔 XML |
| 2 | `{圖名}_{版號}.svg` | 正式圖檔(補白、僅 viewBox) |
| 3 | `{圖名}_{版號}.md` | 流程說明(自動整理,給人讀) |
| 4 | `{圖名}_{版號}_檢視器.html` | 可縮放檢視器(對話內細看,含節點搜尋) |
| 5 | `{圖名}_{版號}_流程定義.py` | 原始定義檔(迭代基準,含版面細節) |
| 6 | `{圖名}_版本記錄.md` | 版本記錄表(**不帶版號**,跨版累積) |

**圖檔 XML 規則**(`emit(x, fmt=...)`):
- **預設 `fmt="drawio"`,不需詢問使用者**;僅當使用者明講要在 bpmn.io
  開啟/編輯時改 `fmt="bpmn"`。
- **兩格式互不相容、不可互餵**:draw.io 原生是 mxGraphModel、不支援 BPMN 2.0 XML,
  `.bpmn` 丟進 draw.io 必然嚴重跑版(非檔案問題)。
- 兩格式座標同源,版面與 SVG 完全一致;`.drawio` 用 draw.io 官方 `mxgraph.bpmn` 形狀。
- 回讀比對:`.bpmn` 用 `bpmn_import.py`;`.drawio` 用 `drawio_import.py`
  (show 重建定義 / diff 差異與建議版號)——**人工微調吸附回格線**:
  挪動的節點以中心點反查 lane/sub、y 反算 row 四捨五入,保留自動佈局能力;
  改名/挪位=文字進次版,增刪節點連線=結構進主版;差異清單經使用者確認才套回。

由 `emit(x, outdir, fmt=..., src=__file__, change=..., change_kind=..., change_source=...)`
一次產齊;定義檔請直接以最終檔名撰寫或交由 `src=__file__` 複製。

**多流程圖專案(單一多頁 .drawio)**:同一案子多張圖時用
`emit_multi([圖1, 圖2, ...], "專案名", outdir, version="V01.00", src=__file__, change=...)`
→ `{專案}_{版號}.drawio`(每圖一頁籤)+ `{專案}_{版號}_檢視器.html`(多頁檢視器,
頁籤切換、各頁獨立縮放/搜尋)+ 各圖各自 SVG/MD + 專案層級版本記錄
(任一圖結構變動→主版進位)。版號以專案為準,所有圖共用。

多張圖時另須遵守:
- **節點 id 加圖前綴**:多頁 .drawio 內節點 id 必須**全檔唯一**,各圖 id
  一律加圖代號前綴(如 `a1_`/`a2_`)。跨頁重複會使 draw.io 連線錯亂、
  箭頭消失;`emit_multi` 已內建偵測、重複即報錯,但撰寫定義時就應加前綴。
- **並行作業、不逐張處理**:多張圖一律在**同一份定義檔**內全部撰寫完成,
  以**單次** `emit_multi` 一次產齊全部交付物,再以單次 `validate_bpmn.py`
  驗證整個資料夾——禁止「畫完一張、交付一張、再畫下一張」的逐一作業
  模式。理解/確認先行階段也一次涵蓋全部圖,微調集中一輪完成後才重跑。

## 首次繪圖流程(確認分級,20260710.12 使用者裁決)

**簡單案直接產出**:輸入明確(單一流程、角色清楚、分支完整)時**不再
先行確認**,直接產出;判讀假設逐條列在交付說明中,事後以迭代修正。
**須確認才產出**的情形:輸入含糊(角色/分支/迴圈出口缺漏,見「資訊
缺漏處理」)、會議記錄含多條流程(先列清單請使用者挑選)、或使用者
明示要先看理解摘要。以下確認先行流程僅適用後者:

1. **理解 INPUT**:來源可為流程說明文字、舊流程圖、**訪談/會議記錄**。
   萃取:角色(→泳道/pool)、步驟(→任務)、決策(→閘道)、平行、
   跨組織訊息、迴圈、系統分區(→bands)。
2. **會議記錄含多條流程時**:先列出偵測到的流程清單(每條附一句摘要),
   **請使用者挑選**要繪製哪些,不要全部直接畫。
3. **確認先行**:產圖前先回報三件事,經使用者確認後才產出——
   - 我理解的流程摘要(角色、主幹步驟、決策點)
   - 不確定的假設(記錄中語焉不詳、我自行補齊的部分)
   - 建議調整處(冗餘步驟合併、缺漏的分支等)
4. **定義與產出**:依 `conventions.md` 撰寫 `_流程定義.py`(起手式省略 row 自動佈局、
   route 極簡:僅回饋迴圈給 `backLoop`),
   `emit()` 產 6 檔;驗證依 `conventions.md`「產出後務必驗證」節,全數通過才交付。
   **版面微調規則(builder 內建五輪自動修正已在 emit 內完成,人工微調成本是
   自動修正的百倍以上)**:
   - **手動微調預設 0 輪**:validate 硬傷(穿節點、連線重合、懸空/孤兒、
     id 衝突)清零即交付;殘餘低權重視覺分**如實回報評分與原因後直接交付**,
     交由使用者在 draw.io 內手動微拉或明示要求後才調——多數殘餘為拓樸極限
     (CHANGELOG 實證),人工重試通常壓不掉。
   - **使用者要求再調時,以 2 輪為硬上限**,且遵守證據式微調:每次改動必
     重跑 `check_layout` 比較評分,變差立即撤回;2 輪未改善即停手回報。
   - **改定義檔一律 str_replace 局部修改,禁止重寫全檔**(大圖重打一次
     即多耗 1–3 分鐘)。
   產出前若涉及 builder 慣例未涵蓋的建模情境(附加事件、事件子流程、
   不確定的閘道取捨、choreography 類元素等),**須先查閱
   `reference/BPMN_2_0_by_Example.md` 對照範例確認建模方式(依檔頭索引定位頁段,
   先讀文字頁段判斷、必要才 view 頁圖且單次至多 2 頁,勿整檔讀入)**,
   確認無誤才進版產出;
   一般泳道/閘道/訊息流案件以 `validate_bpmn.py` 全數通過為合規門檻。

## 效率規則(全流程適用)

- **單次往返原則**:以**一條 bash `cat` 同時併讀必要讀取之檔案**,不分次 view;
  定義檔寫完後 **`python3 定義檔 && python3 scripts/validate_bpmn.py <outdir>`
  併為一次 bash 呼叫**,不分兩步。
- 引擎端到端(含五輪修正與驗證)實測秒級;若整體流程耗時超過數分鐘,
  瓶頸必在往返次數或人工微調輪,回頭檢查是否違反本節與微調規則。

## 資訊缺漏處理(首次繪圖與迭代皆適用)

輸入(流程說明、會議記錄、使用者修改後的圖檔/`.md`/`.bpmn`/描述文字)有缺漏時,
一律在確認先行階段**加列建議調整並詢問**,不得自行補齊後直接產出:

- **角色/泳道缺漏**:先從敘述的動詞主詞推斷角色;列出「推斷角色分道(附假設)」
  與「先以單泳道出圖、後續迭代再分道」兩案,請使用者二擇一。
- **步驟歸屬不明**(不知道哪個角色執行):暫放最可能的泳道,
  在假設清單逐條標明「第 X 步『◯◯』假設由△△執行」,經確認才產出。
- **分支缺漏**(決策只講一邊、流程斷裂、迴圈沒有出口):列入建議調整處,
  提出補接方案或詢問缺漏分支去向;不得留下懸空/孤兒節點
  (`validate_bpmn.py` 會 FAIL,此為第二道防線,不能取代事前詢問)。
- **迭代時**:使用者修改後的檔案若出現上述缺漏(如新增節點未標角色、
  新增決策缺另一分支),同樣併入差異清單後的待確認事項一次詢問。

## 迭代同步流程(使用者修改圖檔、`.md`、`.drawio`、`.bpmn` 或提供描述文字後)

使用者上傳修改後的檔案或描述(最好連同上一版 `_流程定義.py`;新對話請務必請使用者提供)。
依輸入來源決定比對方式:

- **`.md` / `.bpmn`** → 用 `bpmn_import.py` 自動比對(下方第 1 步)。
- **`.drawio`** → 用 `drawio_import.py` 自動比對(下方第 1 步;人工挪動吸附回格線)。
- **圖檔**(標註過的 SVG/PNG、手繪照片等)→ Claude 直接判讀圖面,
  對照上一版逐項列出偵測到的差異;判讀不確定的標註列入不確定處,不要猜。
- **描述文字**(口頭/文字說明要改什麼)→ Claude 整理成逐條變更清單。

圖檔/描述文字來源不經 import 工具 diff,由 Claude 依變更清單自行判斷
「結構/文字」分類,再以 `bump_version()` 提出建議版號。
比對用的**舊基準優先序**:上一版 `_流程定義.py`(執行後 `from_obj()`)
> 上一版圖檔 XML(`.drawio` 或 `.bpmn`,依交付格式)> 上一版 `.md`。

不論來源,後續一律走第 2 步確認先行,`change_source` 對應填
「使用者修改 .md」/「使用者修改 .drawio」/「使用者修改 .bpmn」/
「使用者修改圖檔」/「口頭指示」。

1. **解析與比對**(僅 `.md`/`.drawio`/`.bpmn`,依格式選工具):
   ```bash
   python3 scripts/bpmn_import.py diff <上一版.bpmn 或 .md> <使用者修改後的檔>
   python3 scripts/drawio_import.py diff <上一版.drawio> <使用者修改後的.drawio>
   ```
   或在 Python 內:`bpmn_import.diff(bpmn_import.from_obj(舊物件), bpmn_import.parse(新檔))`。
   舊基準是 `.py` 而使用者改 `.drawio` 時,先執行 `.py` 重產上一版 `.drawio` 再 diff。
   工具會列出差異清單(新增/移除/更名節點、連線與標籤異動、泳道/分區變更)
   並依分類建議下一版號。
2. **確認先行**:把**差異清單 + 不確定處 + 建議版號**回報使用者,確認後才動手。
   解析不到的內容(使用者自創格式、bpmn.io 內新增的元素型別)列入不確定處,不要猜;
   修改後檔案的資訊缺漏依「資訊缺漏處理」一節加列建議調整並詢問。
3. **套用變更**:以上一版 `_流程定義.py` 為基準修改(保留原版面微調);
   若無 `.py` 可用,以 `bpmn_import.py show <檔>`(.bpmn/.md)或
   `drawio_import.py show <檔>`(.drawio)傾印結構重建定義,並註明版面可能微調。
4. **版號進位後重產 6 檔**:`x.version = bump_version(舊版號, structural)`;
   emit 時務必填 `change`(變更摘要)、`change_kind`(結構/文字)、
   `change_source`(「使用者修改 .md」/「使用者修改 .drawio」/「使用者修改 .bpmn」/
   「使用者修改圖檔」/「口頭指示」),
   版本記錄表會自動累積(同版號重跑會覆寫該列,不重複)。
5. **驗證與規格查閱**同首次繪圖第 4 步(含微調規則、效率規則與條件式查例規則)。

## 命名與版號規則

- **圖名**:`Proc` 的 pid/name(`Collab` 的 cid/name 亦同)一律「◯◯**工作流程圖**」
  (例:數位卷證摘要及頁數判讀工作流程圖)。pid/cid 用於**檔名**(檔名安全字元,
  全形符號以 `_` 代替),name 用於**顯示**(SVG 圖頂標題、`.md` 標題),兩者可不同。
- **版號** `V主版.次版`(各兩位數),初版 `V01.00`,附加於檔名、SVG 圖頂標題、`.md` 標題:
  - **結構變動**(節點/泳道/連線增刪、閘道種類變更)→ 主版 +1、次版歸 00(V01.03→V02.00)
  - **僅文字變更**(更名、標籤改字)→ 次版 +1(V01.00→V01.01)
  - 進位交給 `bump_version(v, structural)`,分類依 `bpmn_import.diff` 結果,最終以使用者確認為準。

## 圖別選擇(先判斷)

- **單一組織內、角色協作** → 一個 `Proc`,角色用泳道。
- **跨組織 / 跨單位往來** → `Collab` 多 pool,pool 間用 `message()`。
- **同時進行、全到齊才續** → `kind="parallel"` 平行閘道;擇一用排他。
- **角色之外還要標「系統 / 階段」第二軸** → `Proc(..., bands=...)` 橫向系統分區。
