# 版面與繞線慣例(直式 BPMN)

> 內部機制(佈局試算/選路/分軌/修正輪的演算細節)在 `internals.md`,
> **僅維護 skill 時讀**;產圖讀本檔即可。

## 兩種佈局模式
- **自動佈局(建議起手式)**:`add()` 時**省略 row**,只給 `lane`(角色),連線用預設 route。
  `emit()` 會自動分層、選路並自我修正(多版面試算取優;列數 >12 自動
  加試加寬候選;Collab 於整體層級試算)。手動微調前,先讓它自動排
  一版通常就夠用。**選型加寬(chain/wide)只發生在自動佈局**;手動模式以
  `dx` 微調。**工件泳道加寬則無條件適用**(見「工件同泳道子欄規則」節)。
- **加寬候選共四種,列數 >12 時逐級(2→3 子欄)與單欄比評分取優**:
  - **chain(優先嘗試)**:僅「同泳道、一進一出、非閘道」節點橫向並排;
    任何多連線節點(閘道、匯流點、分岔點)降列自佔一列。適合含長鏈段的流程。
  - **wide(蛇行)**:僅閘道分支降列、其餘一律並排。壓列最兇但長鏈流程易穿線。
  - **chainL / wideL(迴圈親和分欄,20260710.13 學自使用者手修)**:同上兩種,
    但同列混有「迴圈鏈」(能走到 backLoop 回頭邊的節點)與「終止鏈」節點時,
    迴圈鏈固定靠左欄(貼近迴圈通道、鏈內越接近迴圈出口越靠左)、終止鏈固定
    靠右欄,各鏈直下不蛇行——回線就近入左通道,不橫穿終止鏈與自身上游節點。
    適合「決標/廢標」型雙分支一終止一回圈的拓樸(實測 A-1 63→40 分,
    穿節點/重合歸零)。
  - 取優是**按圖各自決定**:同專案可一圖加寬、一圖單欄;評分優者中選,
    不強制加寬(線性長鏈流程單欄直下常是客觀最優,以評分為準)。
- **route 極簡原則**:起手全部交給自動佈局,僅真正的回饋迴圈給 `backLoop`;
  其餘手動 route 只在 `check_layout` 指出具體缺陷時才加,不憑直覺預先指定
  (過度指定會干擾多版面試算,常反而變差)。
  - **例外——雙分支匯流起手式(20260710.04 實戰)**:決標/廢標兩分支各接一段
    後又匯流到同一下游(如都回「招標公告」)這種「菱形→兩鏈→匯流」拓樸,
    自動佈局易在匯流處堆疊交叉/重合。實測**在決標主線(判斷為「是」那條)
    的閘道出邊給一個 `route="sideLeft"` 或 `sideRight` 提示,常讓全局選路翻盤**
    (本輪 A-1 `gw2→決標` 加 sideLeft:評分 103→56)。此為少數值得起手就試的
    提示;仍須 `check_layout` 比分驗證、變差撤回。
- **手動佈局(精修)**:`add(id, type, name, lane, row)` 明確給 `row`,並指定 `route` 細調。
  只要任一**流程節點**省略 row,就會整體改走自動佈局,因此精修時請流程節點
  都給 row;**工件(input/output/note)省略 row 不觸發整體自動**,僅工件
  自動放置(同泳道子欄標準)。

## 資料庫節點(database)佔格規則
- 資料庫(圓柱)以**順序流**串入主線時佔流程格位,排在上下游之間,
  主線直下不繞行;樣式仍為圓柱、語意檢查仍豁免(懸空/孤兒/可達性)。
- 僅以 `assoc()` 關連掛在節點旁時,才視為純工件就近放置。
- 維護提醒:`NONGRID_TS` 已不含 `"database"`,勿加回(加回會使
  順序流串接的資料庫被丟到頂列、主線被迫上繞)。

## 工件同泳道子欄規則(20260710.09 標準)
- **含工件(input/output/note)夥伴的泳道,無條件直接加寬至 ≥2 子欄**
  (不受列數 >12 門檻限制;與選型加寬 chain/wide 疊加時取 max),
  工件佔**夥伴同泳道的相鄰子欄**,不再跨泳道貼鄰。
- 放置優先序:①夥伴同泳道相鄰子欄(**右側優先**,遠離泳道左緣迴圈通道)
  ②夥伴同泳道其他列 ③最後手段才鄰泳道(殘餘距離由「貼鄰」檢核計 3 分回報)。
- 手動佈局亦適用:**僅工件省略 row 不觸發整體自動佈局**,工件自動放置
  (含泳道加寬)照走;工件明給 row/lane 者一律尊重不動(向後相容)。
- 此標準使工件不會落入他泳道左緣的 backLoop 迴圈通道,舊「產出後檢查
  穿節點警告、手工釘 lane」對策已無效移除(歷史見 CHANGELOG 20260710.04/.09)。

## 座標模型
- 角色 = **垂直泳道**(由左到右排列),流程 = **由上而下**(列索引遞增)。
- `lane` 是泳道索引(0 起),`row` 是列索引(0 起)。
- 同一 `row` 不同 `lane` 可並排(例:閘道的「是/否」兩個目標各放一欄,同一列)。

## 避免重疊的規則
1. **主流程走直線**:連續同欄步驟用 `route="straight"`;同列跨欄交接
   自動走側到側單段水平線(不出底邊、不穿節點)。
2. **分支用相鄰欄**:閘道兩分支放左右欄,各走一邊;標籤自動貼在分支線旁(規則見下方閘道節)。
3. **分歧分支(閘道)**:兩分支用 `route="outLeft"`/`outRight`,各自從菱形左/右頂點出發,
   不會從同一點重疊拉出。
4. **回饋/迴圈線**:用 `route="backLoop"`——走**最左側通道**繞回,保證不與分支交錯。
5. **同泳道跳層(中間有節點阻擋)**:用 `route="sideRight"`/`sideLeft`——出側邊、走泳道
   左/右緣通道繞行、再從側邊接入,不穿過中間節點;自動佈局會左右試算選交叉較少的一側。
   `enterRight` 保留給跨欄合流從右側接入的情境。
6. **結束節點就近收尾**:把 end 放在最後一個任務正下方(同欄 `straight`)。
7. **九條鐵則(20260710.12 使用者裁決之產出規格;違規仍交付,但
   emit 輸出與 `.md`「鐵則檢核」區必須顯著標示 ✗ 並附手修建議)**:
   ① 節點(任務/元件/文件)不可互相重疊。
   ② 節點不可壓在泳道線上(邊界事件貼宿主框緣屬 BPMN 慣例,豁免)。
   ③ 連線不可平行重疊、不可穿越節點及文字(標籤/容器標題)。
   ④ 平行線必留人眼可辨間距——**雙門檻(20260710.14)**:短並行(>30px)
   須 ≥10px;**長並行(>150px)須 ≥2×LINE_GAP=24px**(12px 在數百 px 的
   長並行下人眼仍視為重疊,使用者標紅實證)。重合/間距檢核**不因共用
   端點節點而豁免**(舊豁免使同目標雙回線 720px 共線漏檢)。多條
   backLoop 各佔一軌(左通道逐軌外移 2×LINE_GAP)。
   ⑤ 連線**應避免**交錯;避不掉的交叉以**跨線橋**呈現——`.drawio` 用原生
   `jumpStyle=arc`、SVG 由 `_hops_path` 畫半圓弧(後畫者跳線;貼近折點
   8px 內不畫橋)。交叉因此**不列鐵則違規**,列提醒。
   ⑥ 任兩條連線的起迄點不可重疊——選路定案後 `_spread_ports` 自動把
   同節點同側共用端點的邊錯開(僅矩形類節點;圓形事件/菱形閘道錯開
   會浮離外框,不處理,殘餘如實回報)。
   ⑦ 工件避開節點就近擺放、優先同泳道;**所有泳道預設 2 倍 LANE_W 寬,
   單一泳道格點節點 >20 個時 3 倍寬**(`_lane_min_width` 以 lane_pad
   對稱補足,節點居中、兩側留走線通道)。
   ⑧ 連線端點取靠近/鄰近端(端點選擇總則,見第 10 條);僅為避免
   交錯才改走較佳端點/路徑。
   ⑨ 不交錯前提下走最短距離、最少轉折;為避交錯/重疊而增加時取最小。
   引擎機制(選路二輪精修+至多五輪修正)見 `internals.md`;
   每張圖引擎時間上限 5 分鐘(`ENGINE_TIME_CAP`),emit 如實回報耗時。
8. **邊界事件出邊自動走「讓位軌」(bndSide)**:沿泳道邊緣夾縫繞行、
   避開宿主下方扇出,幾何細節見 `internals.md`;不需也不應手動指定。
9. **一律正交、不出現斜線**:所有連線都是水平/垂直正交折線(`straight` 跨欄時自動轉正交);
   `check_layout` 與 `validate_bpmn.py` 皆會偵測斜線並回報。
10. **端點選擇總則(最近端優先 → 鄰近端 → 禁對向端)**:每個節點有 top/bot/left/right
    四個接點。連線起訖點的選擇優先序:
    - **最近端優先**:朝目標方向的那一面出/入(目標在下走 `bot→top`、在右走
      `right→left`);`_orth_down` 已依目標相對位置自動選最近端對。
    - **次為鄰近端**:最近端會造成穿越/重疊時,退用相鄰面(如改走側邊
      `sideLeft`/`sideRight`、`enterRight` 從右接入)。
    - **不可用對向端**:絕不從背對目標的那一面拉出(目標在下卻從 `top` 出、
      目標在右卻從 `left` 出)。**理由(使用者確認之因果)**:用對向端會強迫連線
      繞過節點本體才能轉向目標,兩害必居其一——要嘛為避節點而繞成非最短路徑
      (觸發「繞行過長」),要嘛走最短卻反貫穿自身節點(觸發「連線穿越節點」)。
      故對向端是穿越與繞行的**共同上游**,從端點選擇階段就排除,比事後修繞線更根本。
    - 此總則已隱含在 route 引擎(`_orth_down` 選近端、side/enter 系列選鄰近端);
      本條將其顯性化為規範,供首次產製與維護援引。

## 閘道種類與多 pool 協作
- **閘道 kind**:`add(..., "gateway", ..., kind=)` 可選 `exclusive`(預設,擇一)/`parallel`
  (`+`,同時分岔、合流等全部到齊)/`inclusive`(`O`,條件為真者皆走)。平行/包容閘道
  通常不必命名(符號即語意);排他閘道分支建議標條件。
- **排他閘道名稱置於菱形內、菱形固定尺寸不放大**:名稱 **≤3 字單行**(如「達標?」),
  **4 字自動折兩行**(2字/行,如「面談方式」→「面談/方式」),**≥5 字**會溢出菱形,
  `check_layout` 會自動警告。詳細判斷準則寫在閘道前的「判斷任務」,不塞進閘道名稱。
- **分支標籤貼在線旁**:優先置於分支岔開後的水平段上方(65% 處、遠離菱形頂點);
  無水平段時貼在垂直段右側,不會退到菱形頂點。
- **判斷語意必須自足(首次產製即到位)**:菱形名稱受 ≤4 字限制,但**分支標籤不受限**——
  讀圖者要能只看菱形+分支就懂「在判斷什麼、什麼結果走哪」。三條要求:
  - **菱形用疑問短語**:如「有無投標」「達底價?」,不要用動作簡稱(「進入底價」
    「接受議約」讓人不知是判斷還是任務)。裝不下的完整條件寫在**閘道前的判斷任務**。
  - **分支標籤要具體、成對**:排他閘道兩出邊都要標,且用能對應菱形問題的詞——
    「達底價→是/否」而非光「是/否」;「有無投標→有/無」需與菱形問題語意呼應。
  - **判斷任務前置**:凡條件複雜(如「是否達底價且廠商同意議約」)須在閘道前放一個
    task 描述判準,閘道只做路由(對應建模原則:決策在 task 完成、閘道只路由)。
- **多 pool 協作(Collab)**:跨組織/跨單位往來時,每個參與者用一個 `Proc` 當 pool,
  以 `add_pool()` 加入,pool 會**由左到右並排**;用 `message(src, tgt, label)` 連接兩個
  **不同 pool** 的節點,產出虛線 message flow(起點空心圓、開口箭頭)。
- 各 pool 共用同一列(row)格線,訊息流多為水平,易讀;pool 間距(POOL_GAP)自動留給訊息流。
- 節點 id 在整個 Collab 內須唯一(跨 pool 也是)。
- **XML 內的 id 一律 ASCII**(字母/底線開頭,限 `[A-Za-z0-9_.-]`),**中文只放 name**:
  bpmn.io 的 ID 驗證是 ASCII NCName,中文 id 會匯入失敗(illegal ID)。
  - `pid`/`cid` 為雙軌:原字串(可中文)用於**檔名與顯示**;XML 內改用自動清洗的
    `xid`(`_ncname()`:非法字元轉底線 + md5 前 6 碼保唯一、同輸入必同輸出)。
  - 節點 id(`add`/`flow`/`message`/`bands`)在建模時即清洗,引用自動一致;
    但撰寫流程定義時節點 id 仍應直接用英文,可讀性最好。
  - **id 不可用 JS 內建屬性名**(`fill`/`map`/`push`/`filter`/`sort`/`find`/
    `keys`/`values`/`join`/`length`/`toString`/`constructor` 等 Object/Array
    prototype 成員):draw.io 以純 JS 物件/陣列作 id 查找表,撞名時開檔即報
    「setId is not a function」(20260710.10 桌面版實測)。`_ncname()` 已自動
    將撞名者改為「原名_md5前6碼」(如 `fill`→`fill_a552c7`)、
    `validate_bpmn.py` 亦會攔截(第二道防線);**撰寫定義時仍應直接避開**
    (如用 `fill_form`、`sort_docs`),保持 .py 與圖檔 id 一致、迭代 diff 乾淨。

## 版面可讀性評分與檢核項目(權重單一來源)
分數越低越好、0 = 無缺陷;只在「同一張圖的不同版面變體」之間比較才有意義。
調整 row / route 後重跑比較分數,即可量化判斷版面是否變好。
`bpmn_builder.layout_score(check_layout(x))` 計全部類別;
`validate_bpmn.py --score` 只計自身可偵測的類別(權重表共用、涵蓋範圍不同)。

| 檢核項目 | 分 | 判準 |
|---|---|---|
| 連線穿越節點 | 20 | 連線段貫穿節點內部 |
| 斜線 | 15 | 非水平/垂直的連線段 |
| 連線交叉 | 10 | 兩條連線相交(拓樸無法避免時可接受並說明) |
| 連線重合 | 8 | 共線重疊且分軌隔不開 |
| 節點重疊 | 5 | 節點框互相重疊 |
| 節點壓泳道線 | 5 | 節點超出所屬泳道邊界(鐵則②;邊界事件豁免) |
| 端口重合 | 5 | 同節點多條線共用同一進出點 |
| 平行間距不足 | 5 | 雙門檻(鐵則④):短並行(>30px)距離 <10px;長並行(>150px)距離 <2×LINE_GAP=24px |
| 框線重疊 | 5 | 連線沿泳道/pool/容器框貼行(≤4px 貼行 >30px;交錯穿越不算) |
| 分區列範圍重疊 | 5 | 兩個 bands 分區的列範圍交錯(版面試算與修正輪會優先避免;分區成員與流向交錯時屬建模問題,計分後如實回報) |
| 繞行過長 | 3 | 路徑長 > 曼哈頓距 ×2.2 且超出 >150px |
| 轉折過多 | 3 | 轉角數超出基準(最短最少彎原則) |
| 容器標題壓線 | 3 | 連線段壓在容器名稱文字上 |
| 工件離夥伴過遠 | 3 | 工件離關連夥伴格距 >1 |
| 標籤碰撞/溢出 | 3 | 標籤壓線或壓到節點、標籤超出 pool 邊緣、閘道名 ≥5 字溢出菱形等 |
| 順序流缺方向 | 硬性 | 順序流(實線非關連/訊息流)未顯示箭頭方向。**屬檔案級硬傷、不計權重分**:由 emitter 保證輸出 `endArrow=block`、`validate_bpmn.py` 座標再驗攔截(20260710.04)。check_layout 不評此項(幾何評分器不檢查樣式字串) |

訊息流跨 pool 飛越節點屬慣例,列提醒級不計分。

> **評分器涵蓋範圍**:上表所有帶權重的項目即 `check_layout` / `layout_score`
> 的計分類別(權重單一來源;程式內「標籤」「溢出」兩鍵在表中合併為
> 「標籤碰撞/溢出」列、「工件離夥伴過遠」對應訊息鍵「貼鄰」);
> 「順序流缺方向」為檔案級硬性項,只在 `validate_bpmn.py`
> 攔截,不進 `layout_score` 加權(它驗的是輸出樣式而非幾何)。三方帳目:
> conventions 登記全部 → check_layout 計全部權重項 → validate 計幾何子集
> (穿越/斜線/交叉/重疊)+ 箭頭方向硬性項。

## 語意檢核項目(check_semantics,emit 必跑;不計分、逐項回報)
- **起訖**:每個 pool 須有 start 與 end 事件。**豁免:`terminate` 視為合法
  終點;無 incoming 的 `message`/`timer` 視為觸發起點**——維護時勿把這
  兩條豁免當 bug 修掉。
- **懸空/孤兒**:節點無下一關=懸空、無上一關=孤兒。**豁免:工件與
  文字註解不參與語意檢查**(懸空/孤兒/可達性皆免)。
- **可達性**:每個節點須從起點可達、且能走到任一終點。
- **閘道**:排他閘道分支未命名 → 建議級提醒(建議標條件);
  閘道單進單出 → 「通常多餘」提醒(對應建模原則:非關鍵決策不設閘道)。
- **訊息流**:端點須存在,且兩端須在**不同 pool**。

## BPMN 2.0 合規重點(本函式庫已自動處理)
- 每個 flow node 自動補 `<incoming>`/`<outgoing>`,且 incoming 先於 outgoing。
- `collaboration/participant` + `process/laneSet`;`BPMNPlane` 指向 collaboration。
- 垂直泳道 `isHorizontal="false"`;每個節點/連線都有對應 DI(含 ≥2 個 waypoint)。

## 組織 BPMN 繪製規範(圖例落地)
配色採 draw.io 標準色盤,單一來源在 `bpmn_builder.STYLE`(三種輸出統一讀取;

## 三格式檢核對照(單一來源分層)
| 檢查層 | 位置 | 涵蓋 | 適用 |
|---|---|---|---|
| 語意 | `check_semantics(x)`(emit 必跑) | 懸空/孤兒、start/end 數量、可達性、閘道規則、訊息流跨 pool | **兩種 fmt 皆同** |
| 版面 | `check_layout(x)`(emit 必跑) | 重疊、穿框、交叉、重合、斜線、標籤碰撞、閘道字數、bands | **三種輸出共用同一幾何** |
| 檔案級 .bpmn | `validate_bpmn.py` | NCName、參照、incoming/outgoing、泳道覆蓋、DI 完整、**交付完整性**(必產檔齊備;可選檔缺列提醒)+ 語意/版面**再驗**(第二道防線,涵蓋使用者手改檔) | .bpmn |
| 檔案級 .drawio | `validate_bpmn.py` | 根節點 0/1、id 唯一、參照、mxGeometry、**交付完整性** + 幾何**再驗**(重疊/穿框,排除框架 cell) | .drawio |
| 檔案級 .svg | `validate_bpmn.py` | 預覽安全(viewBox/比例/背景) | .svg |

## 產出後務必驗證(本環境無法 render SVG,故以下兩步為主)
1. **離線版面檢查**:`emit()` 會自動跑 `check_layout(p)`,在終端列出「節點重疊 / 連線穿過節點 /
   連線交叉」。也可單獨呼叫 `check_layout(p)` 取得問題清單。是否人工調整依
   `workflow.md` 微調規則:**硬傷清零即交付,殘餘低權重視覺分如實回報、預設 0 輪手動微調**。
2. **結構與版面驗證**:跑 `python3 scripts/validate_bpmn.py <資料夾>`,檢查參照一致性、
   懸空/孤兒、起點可達性、能否抵達終點、閘道分支、DI 完整、**交付完整性
   (必產 3 檔缺=FAIL;可選檔缺=提醒)**,
   並從圖檔 XML(.bpmn 或 .drawio)的座標再驗一次重疊/穿框。
   未產圖檔 XML 時本步無檔可驗,以第 1 步 emit 內建檢核為交付門檻。
3. **真要看圖**:把 `.svg` 用瀏覽器開、或把 `.bpmn` 拖進 [bpmn.io](https://bpmn.io) 檢視。
   注意:本執行環境的 ImageMagick **缺 SVG delegate,`convert x.svg x.png` 會失敗**,
   且通常無網路,因此請依賴上面的離線檢查,不要靠 ImageMagick 轉圖。

## 建模原則補充
- **決策在 task 完成,閘道只做路由**:每個排他閘道前通常要有一個「判斷任務」;需要對照規則時查 `reference/BPMN_2_0_by_Example.md`(依檔頭索引定位,勿整檔讀入)。
- **非關鍵決策不設閘道,收成一個任務**:閘道保留給「走不同路、有不同後果」的關鍵分岔
  (如審核通過/退回);殊途同歸的操作細節(如「複製或匯入至某系統」)寫在同一個任務名稱內即可,
  避免多佔欄位、徒增節點而不增加資訊。
- 輸出的 `.md` 由流程定義自動整理(各 pool 的角色、步驟、決策點、連線、跨 pool 訊息流),不需手寫。
- **SVG 根元素只給 `viewBox`、不寫死 `width`/`height`**:預覽才能縮放;bpmn.io / Figma 讀的是 viewBox 座標系,不受影響。

## 橫向系統分區(bands)
- 需求:垂直泳道已表達「角色」,另想以第二軸標示「系統/階段」。BPMN 2.0 泳道僅單軸,
  規格內最接近的元素是 **Group 人工產物**(虛線框、跨泳道視覺分組、不影響流程語意)。
- 用法:`Proc(pid, name, lanes, bands=[("系統A", ["s","t1"]), ("系統B", ["t2","e"])])`,
  每個分區給名稱與所屬節點 id,帶區範圍由節點列自動計算,顏色依內建色盤輪替。
- 產出效果:`.bpmn` 寫入標準 `category + group + DI`(bpmn.io 以虛線框+名稱呈現、可編輯);
  SVG 於泳道區鋪淡色帶與虛線分隔,pool 左側直欄改為分區標頭欄(直排名稱);
  `.md` 自動產生「橫向系統分區」章節。
- 有 bands 時 pool 名稱**僅置於圖頂標題**,不再於左側直條重複出現。
- 檢核:分區**引用不存在的節點**列缺陷;兩分區**列範圍交錯**計 5 分(見檢核表)。

## viewBox 自適應與預覽安全(重要)
- viewBox 依內容實際範圍計算,四邊各留 24px 小留白;**比例不寫死**,偏長或偏寬皆隨內容。
- 寬高比 < 1.3 的直式細長圖自動左右補白至 1.3;補白**封頂於內容寬 × 2**
  (主圖至少佔畫布一半),封頂生效時 SVG 根元素標 `data-pad="capped"`——
  對話內預覽可能裁底,**請引導使用者以 `_檢視器.html` 細看**。
- `validate_bpmn.py` 對所有輸出 `.svg` 檢查:①不寫死 width/height
  ②寬高比 ≥1.3(帶 `data-pad="capped"` 者放行、改列 • 提醒)③背景覆蓋完整。

## HTML 檢視器(可選交付物,使用者勾選才產)
- 使用者勾選時由 `emit(viewer=True)` 產出 `_檢視器.html`;**不要手寫檢視器**(內嵌與縮放機制見 `internals.md`)。
- 分工:`.svg`(補白、僅 viewBox)為正式交付與 bpmn.io / Figma 匯入用途;
  `_檢視器.html` 僅供對話內縮放檢視,其內嵌 SVG 不受 `.svg` 預覽安全規則約束。
- **多頁檢視器 defs id 隔離(維護提醒)**:多張 SVG 內嵌同一 HTML 時,
  各頁 defs 內 id(箭頭 marker 等)自動加頁碼前綴(`p0_`、`p1_`…)並同步
  改寫 `url(#…)`/`href="#…"` 引用——id 跨頁重複時瀏覽器一律解析到
  文件第一個定義,該頁 `display:none` 時其餘頁箭頭全部消失。
  此為 `build_viewer_html_multi` 內建行為,勿移除。
